Opaque Secret

要求 value 必须是 base64 编码的

测试：如果 value 不是 base64 呢？

准备如下 yaml 用来创建 secret，其中 value 不是 base64 格式的：

secret_not_base64.yaml

apiVersion: v1
kind: Secret
metadata:
  name: mysecret-not-base64
type: Opaque
data:
  user: root
  pass: 123r

运行并查看：

$ kubectl apply -f secret_not_base64.yaml

$ kubectl get secret                           
NAME                  TYPE                                  DATA   AGE
mysecret-not-base64   Opaque                                2      42m

$ kubectl get secret mysecret-not-base64 -oyaml
apiVersion: v1
data:
  pass: 123r
  user: root
kind: Secret
metadata:
  annotations:
    kubectl.kubernetes.io/last-applied-configuration: |
      {"apiVersion":"v1","data":{"pass":"123r","user":"root"},"kind":"Secret","metadata":{"annotations":{},"name":"mysecret-not-base64","namespace":"default"},"type":"Opaque"}
  creationTimestamp: "2022-10-10T14:25:30Z"
  name: mysecret-not-base64
  namespace: default
  resourceVersion: "158457"
  uid: 44ae7fda-3f7f-4da3-95b1-c3068727d765
type: Opaque

貌似没什么影响啊？可以正常创建，也可以查看其中的值，继续测试看看

该 yaml 从名为 mysecret-not-base64 的 secret 中引用值，并将其设置为环境变量，然后会执行 env 命令查看当前系统里的所有环境变量（busybox 是一个集成了一些 linux 同样命令的镜像）

secret_not_base64_pod.yaml

apiVersion: v1
kind: Pod
metadata:
  name: secret-not-base64-pod
spec:
  containers:
  - name: secret1
    image: busybox
    command: [ "/bin/sh", "-c", "env" ]
    env:
    - name: USERNAME # 设置名为 USERNAME 的环境变量，其值引用 mysecret-not-base64 中的 user 字段
      valueFrom:
        secretKeyRef:
          name: mysecret-not-base64 
          key: user
    - name: PASSWORD
      valueFrom:
        secretKeyRef:
          name: mysecret-not-base64 
          key: pass

运行并查看：

$ kubectl apply -f secret_not_base64_pod.yaml 

$ kubectl logs  secret-not-base64-pod

USERNAME=��-

发现我们设置的两个环境变量 USERNAME 和 PASSWORD 的值都显示为乱码，莫非如果引用了Opaque 类型的 secret 中的字段，会执行解码操作，将解码出来的值作为 value 进行设置吗？

为了搞清楚这个问题，我们可以再尝试做一个 value 是 base64 编码过的 secret：

secret.yaml

apiVersion: v1
kind: Secret
metadata:
  name: mysecret
type: Opaque
data:
  user: YWRtaW4=
  pass: MWYyZDFlMmU2N2Rm

tips：linux/mac 自带了 base64 命令，可以方便的获取值的 base64:

$ echo -n "admin" | base64
YWRtaW4=

$ echo -n "admin321" | base64
YWRtaW4zMjE=

$ echo "YWRtaW4=" | base64 -d    
admin

# 解密
$ echo "MWYyZDFlMmU2N2Rm" | base64 -d
1f2d1e2e67df

secret_base64_pod.yaml

apiVersion: v1
kind: Pod
metadata:
  name: secret-base64-pod
spec:
  containers:
  - name: secret1
    image: busybox
    command: [ "/bin/sh", "-c", "env" ]
    env:
    - name: USERNAME
      valueFrom:
        secretKeyRef:
          name: mysecret 
          key: user
    - name: PASSWORD
      valueFrom:
        secretKeyRef:
          name: mysecret 
          key: pass

运行：

$ kubectl apply -f secret_base64_pod.yaml  
pod/secret-base64-pod created

$ kubectl logs secret-base64-pod
KUBERNETES_SERVICE_PORT=443
KUBERNETES_PORT=tcp://10.96.0.1:443
HOSTNAME=secret-base64-pod
SHLVL=1
HOME=/root
USERNAME=admin
KUBERNETES_PORT_443_TCP_ADDR=10.96.0.1
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
KUBERNETES_PORT_443_TCP_PORT=443
KUBERNETES_PORT_443_TCP_PROTO=tcp
KUBERNETES_SERVICE_PORT_HTTPS=443
KUBERNETES_PORT_443_TCP=tcp://10.96.0.1:443
KUBERNETES_SERVICE_HOST=10.96.0.1
PWD=/
PASSWORD=1f2d1e2e67df

发现环境变量 USERNAME 和 PASSWORD 的值都正常显示了，看来确实是这样，当引用Opaque 类型的 secret 中的值时，会自动做 base64 解码操作

再试试用挂载的方式：

这里将 mysecret 这个 secret 下的所有条目挂载到了容器内的 /projected-volume 目录下，mysecret 就是上面定义的 secret.yaml：

secret-test-pod.yaml

apiVersion: v1
kind: Pod
metadata:
  name: test-projected-volume 
spec:
  containers:
  - name: test-secret-volume
    image: busybox
    args:
    - sleep
    - "86400"
    volumeMounts:
    - name: mysql-cred
      mountPath: "/projected-volume"
      readOnly: true
  volumes:
  - name: mysql-cred
    projected:
      sources:
      - secret:
          name: mysecret

运行并查看效果：

$ kubectl apply -f secret-test-pod.yaml  
pod/test-projected-volume created

$ kubectl get po                         
NAME                                READY   STATUS    RESTARTS   AGE
test-projected-volume               1/1     Running   0          7s

$ kubectl exec -it test-projected-volume -- cat /projected-volume/pass                              
1f2d1e2e67df%                                                                                                                                                             $ kubectl exec -it test-projected-volume -- cat /projected-volume/user
admin%

发现这里将 secret 中的每个条目都单独做成了一个文件，并且文件内容都是已经解码过的值

再试试挂载引用之前那个 value 不是 base64 的 secret：

apiVersion: v1
kind: Pod
metadata:
  name: test-projected-volume-not-base64
spec:
  containers:
  - name: test-secret-volume
    image: busybox
    args:
    - sleep
    - "86400"
    volumeMounts:
    - name: mysql-cred
      mountPath: "/projected-volume"
      readOnly: true
  volumes:
  - name: mysql-cred
    projected:
      sources:
      - secret:
          name: mysecret-not-base64 # 修改这里

运行：

$ kubectl exec -it test-projected-volume-not-base64 -- cat /projected-volume/user
??-%                                                                                                                                                                      $ kubectl exec -it test-projected-volume-not-base64 -- cat /projected-volume/pass
?m?%

发现乱码了
