Istio Bookinfo

2023-05-27
7分钟阅读时长

前言

Bookinfo 是 istio 官方提供的一个演示应用，并且官方也给出了详细的功能描述：

这个应用模仿在线书店的一个分类，显示一本书的信息。 页面上会显示一本书的描述，书籍的细节（ISBN、页数等），以及关于这本书的一些评论。

Bookinfo 应用分为四个单独的微服务：

  • productpage. 这个微服务会调用 details 和 reviews 两个微服务，用来生成页面。
  • details. 这个微服务中包含了书籍的信息。
  • reviews. 这个微服务中包含了书籍相关的评论。它还会调用 ratings 微服务。
  • ratings. 这个微服务中包含了由书籍评价组成的评级信息。

reviews 微服务有 3 个版本：

  • v1 版本不会调用 ratings 服务。
  • v2 版本会调用 ratings 服务，并使用 1 到 5 个黑色星形图标来显示评分信息。
  • v3 版本会调用 ratings 服务，并使用 1 到 5 个红色星形图标来显示评分信息。

kind: Gateway
metadata:
  name: bookinfo-gateway
spec:
  selector:
    istio: ingressgateway # use istio default controller
  servers:
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "*"
---

kind: VirtualService

    - uri:

    - uri:

这个 yaml 里面包括了Gateway （注意是其 apiVersion 是 istio，不是 k8s 原生的 Gateway 资源 ）和 VirtualService 两种资源，下面来简单说明一下这两个资源。

Gateway

Gateway 在网格边缘接收外部访问，并将流量转发到网格内的服务。Gateway 一般需要和 VirtualService 配合使用，其中 Gateway 定义了服务怎样从外面访问，而 VirtualService 定义了请求进入到内部后，其流量如何流转。

在这个 yaml 里，Gateway 包含了 selector 和 servers 两个字段，下面对这两个字段的作用进行一下记录。

selector

Gateway 的 spec.selector 字段用于指定入口处运行了 Envoy 的 Pod，这些 Pod 会执行 Gateway 定义的规则。在 bookinfo 里指定的是istio: ingressgateway ，即所有拥有 istio: ingressgateway label 的 Pod。这些 Pod 由 istio-system namespace 下的名为 istio-ingressgateway 的 Deploy 创建，可以看一下这个 Deploy 的定义：

k describe -n istio-system deploy istio-ingressgateway

spec:
  template:
    metadata:
    # .... 省略
      labels:
        app: istio-ingressgateway
        istio: ingressgateway # 这里
    spec:
      image: docker.io/istio/proxyv2:1.17.2
      imagePullPolicy: IfNotPresent
      name: istio-proxy

可以看到在 label 这里定义了这个 istio: ingressgateway label，同时还注意到其使用的镜像是 docker.io/istio/proxyv2:1.17.2，docker.io/istio/proxyv2 是 Istio 中的 Envoy 代理镜像，表明其创建的 Pod 运行的都是 Envoy，用于接收外部请求并将其转发到 Istio 网格中的适当服务。

VirtualService

VirtualService 定义了对特定目标服务的一组流量规则，即满足什么条件的流量被哪个后端处理。

先来看看 VirtualService 有哪些字段

hosts

这里的 hosts 定义貌似和 Gateway 中某个 servers 中定义的 Host 有关系，会判断二者的 hosts 是否匹配，如果匹配，则代表当你访问该 Gateway servers 时，会进一步走 VirtualService 定义的流量规则，为了验证这个猜想是否正确，可以手动实践一下：

首先需要在 /etc/hosts 中配置一下我们将要添加的域名，其中 IP 从 istio-ingressgateway 这个 service 中获得，因为我使用的是 NodePort 类型的，所以这里的 IP 可以设置为集群内任意一个节点的 IP：

echo 192.168.31.108 test.com >> /etc/hosts

修改 Gateway Resource 的声明：

k edit gw -n meshapp bookinfo-gateway

添加一个 server，声明如下所示：

spec:
  selector:
    istio: ingressgateway
  servers:
  - hosts:
    - '*'
    port:
      name: http
      number: 80
      protocol: HTTP
  # 新添加的 server
  - hosts:
    - test.com
    port:
      name: http1
      number: 9527
      protocol: HTTP

这里我添加的新的 server 将通过 test.com 作为 host 来访问（似乎必须这样，如果用 IP 访问会 404），当然还要加上对应的端口号（比如 http://test.com:31225/login），这里我定义为 9527 这个端口。

然后我们再添加一个新的 VirtualService ：

apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: bookinfo1
spec:
  hosts:
  - test.com # 与 gateway 新添加的 server 的 hosts 相匹配
  gateways:
  - bookinfo-gateway
  http:
  - match:
    - uri:
        exact: /productpage
    - uri:
        prefix: /static
    - uri:
        exact: /login
    - uri:
        exact: /logout
    - uri:
        prefix: /api/v1/products
    route:
    - destination:
        host: productpage
        port:
          number: 9080

修改 service，新暴露一个端口，让其指向我们新创建的 Gateway server 的 port（即 targetPort），也就是 9527，并且该 server 也监听在 9527 端口，然后通过 NodePort 31225 暴露给外界。

- name: http3
  nodePort: 31225
  port: 9527
  protocol: TCP
  targetPort: 9527

现在我们就可以尝试通过 serviceIP + nodePort 来访问这个新定义的服务了：

curl http://test.com:31225/productpage

<!DOCTYPE html>

可以看到我们可以成功访问这个 server。
