两个 main goroutine 操作一个 channel 会怎样

2021-08-08
1分钟阅读时长

引子

定义一个全局变量 chan，然后创建两个 main goroutine，一个往全局 chan 里发送数据，一个从全局 chan 里读取数据，会发生什么？

这个奇葩问题是我在写一个 server 项目时遇到的，当时的情景是：想为 server 添加一个心跳检测功能，客户端每隔一段时间就向服务端发送一个心跳包，服务端这边维持一个定时器，如果收到心跳包则重置，如果定时器到时则认为客户端已断开，关闭 socket 连接。考虑到心跳包产生的网络传输开销，突发奇想，可不可以用一个全局 channel 来代替？

于是便定义了一个全局 channel，客户端每隔一段时间就向 channel 中写入数据，服务端使用 select 来读取 channel。但是测试时却发现，客户端处的发送语句 ch <- struct{}{} 被阻塞了，于是又将全局 channel 改为带缓冲的，缓冲为 1，此时再次运行，客户端的发送语句没有被阻塞了，但是服务端那边却压根收不到数据。

情景再现

创建一个 global 包，在其下新建一个 go 文件，内容如下：

package global

var Ch = make(chan int64, 1)

创建一个 cli 包及其之下的 go 文件：

package main

import (
	"fmt"
	"gomodtest/sockchan/global"
	"time"
)

func main() {
	select {
	case <-time.After(time.Second * 8) :
		fmt.Println("timeout !")
	case v := <-global.Ch:
		fmt.Println(v)
	}
}

最后创建一个 sev 包：

package main

import (
	"gomodtest/sockchan/global"
	"fmt"
)

func main() {
	fmt.Println("send data to global chan")
	global.Ch <- 1
	fmt.Println("send ok")
}

目录结构：

├── cli
│   └── cli.go
├── global
│   └── var.go
└── sev
    └── sev.go

先运行 cli.go，再开一个 termin 运行 sev.go：

sev.go 输出如下：

➜  sev go run sev.go
send data to global chan
send ok

cli.go 输出如下

➜  cli go run cli.go
timeout !

结果说明，可以向 chan 中写入数据，但是不能从 chan 中读取数据。

原因

两个 main 代表两个进程，进程间通信当然不能使用 channel 了（如此弱智的问题）
